# Maintainer Tutorial

This document is for a professional software engineer who is new to Rust, new to Wayland, and new to this level of systems-facing desktop work.

If you work through it, you should come away able to maintain this repository without treating it as magic.

You do not need to do the interactive parts to get value from the document. They are there so you can verify your understanding when you want to.

## What This Repo Is

`gnome-randr` is a GNOME Wayland display-management CLI.

It is not talking to RandR.

It is talking to Mutter over D-Bus, using the `org.gnome.Mutter.DisplayConfig` interface.

That distinction explains most of the architecture:

- query state by asking Mutter for the current display configuration
- change state by sending a full monitor configuration back to Mutter
- use typed native controls when Mutter exposes them
- do not invent fake `xrandr` compatibility where the backend does not support it

The project aims for practical `xrandr` capability parity where that maps cleanly to GNOME Wayland. It does not aim for RandR internals or syntax compatibility.

## Using `gdctl` As The Native Reference

`gdctl` is the upstream Mutter CLI for the same backend.

That alignment work is complete for the current backend. The repo now has a
stable policy: treat `gdctl` as the native semantic reference, but keep
`gnome-randr` as the higher-level operational CLI.

For maintenance, treat it as the semantic reference for:

- native enum names
- native capability boundaries
- which controls are genuinely backend-native versus higher-level convenience
- what Mutter itself considers supported user-facing surface area

Do **not** treat it as a command-syntax compatibility target.

This repo intentionally keeps a different CLI shape:

- `query` instead of only `show`
- `modify` instead of only `set`
- `apply FILE` for saved-layout restore
- software brightness and gamma on top of the native backend surface
- `query --json` for scripting

There is now also a small additive alias layer where the mapping is exact:

- `show` is an alias for `query`
- `set` is an alias for `modify`
- `--verify` is an alias for `--dry-run` on `modify` and `apply`
- remaining typed gdctl monitor-property parity now includes `sdr-native` color mode and `rgb-range`
- monitor leasing now maps to `modify --for-lease-monitor CONNECTOR...`

The native-control surface that should now stay aligned with `gdctl` includes:

- layout mode
- transforms expressed as rotation plus reflection
- color mode, rgb range, and monitor leasing
- power-save, backlight, and luminance

Those aliases are intentionally narrow. They make the native correspondence easier to see, but they do not turn this repo into a `gdctl` syntax clone.

Maintenance rule:

- align to `gdctl` on model and semantics first
- only align syntax when the mapping is clean and non-disruptive
- make intentional divergences explicit in docs instead of leaving them implicit

If you are unsure whether a new feature belongs in this repo, ask two
questions:

1. Does `gdctl` or Mutter already expose a typed native concept for it?
2. Is `gnome-randr` adding value by wrapping that concept in a clearer or more scriptable workflow?

If the answer to the first question is no, be suspicious. If the answer to the
second question is also no, it probably does not belong here.

## The Core Mental Model

There are three important state layers in this repo:

1. Raw D-Bus API surface
2. Typed Rust wrappers around that surface
3. CLI behavior and policy

Keep those layers separate in your head.

### Layer 1: Raw D-Bus

`src/display_config/raw.rs` is autogenerated code.

Treat it as a binding layer, not hand-designed application code.

It exposes the raw Mutter interface:

- `GetCurrentState`
- `ApplyMonitorsConfig`
- `GetResources`
- `SetBacklight`
- `SetLuminance`
- `ResetLuminance`
- power-save property access
- gamma ramp methods

Maintenance rule:

- avoid editing `raw.rs` unless the generated API no longer matches the runtime interface or newer dependencies forced small compatibility fixes
- prefer wrapping it in `proxied_methods.rs` instead of spreading raw tuples and `PropMap` usage through the CLI

### Layer 2: Typed DisplayConfig Wrappers

`src/display_config/proxied_methods.rs` and sibling files turn raw tuples and `PropMap`s into typed Rust structures.

This is where the repo defines the stable internal vocabulary:

- `DisplayConfig`
- `LogicalMonitor`
- `PhysicalMonitor`
- `Resources`
- `ApplyConfig`
- `ApplyMonitor`
- `ColorMode`
- `PowerSaveMode`
- `Gamma`
- `GammaAdjustment`
- native state types such as `NativeDisplayState`

If you are adding a backend capability, this is usually the right place to add the typed representation first.

### Layer 3: CLI Policy

`src/cli/` decides how to expose backend capabilities to users.

This is where the repo answers questions like:

- what flags should exist?
- what combinations are invalid?
- what should be a dry-run preview versus a real apply?
- which unsupported things should fail clearly rather than being emulated badly?

The CLI layer is intentionally opinionated.

## The Most Important Wayland/Mutter Concepts

### Physical Monitor vs Logical Monitor

This is the single most important conceptual split in the repo.

A physical monitor is the actual output identity:

- connector
- vendor
- product
- serial
- supported modes
- per-monitor properties

A logical monitor is the layout object Mutter applies:

- `x`, `y`
- `scale`
- `transform`
- `primary`
- one or more associated physical monitors

If multiple physical monitors are mirrored, they are represented as one logical monitor with multiple associated connectors.

That is why cloning is implemented by merging connectors into one logical monitor, not by stacking separate logical monitors at the same coordinates.

### Layout Mode

The repo now supports these layout modes:

- `logical`
- `physical`
- `global-ui-logical`

These are Mutter-native concepts.

They change how monitor geometry should be interpreted and applied.

If you change layout math, always check whether you are doing geometry in logical coordinates or physical coordinates.

### Rotation vs Reflection

Mutter stores both in one packed transform value.

The repo exposes them as separate typed concepts for users:

- rotation: `normal`, `right`, `inverted`, `left`
- reflection: `normal`, `x`, `y`, `xy`

Internally, these are recomposed into the packed transform bits.

Maintenance risk:

- transform decomposition is lossy in some cases unless you choose a canonical interpretation
- this repo intentionally uses a consistent canonical decomposition rather than pretending the original authoring path is recoverable from the packed bits alone

### Software Color vs Native Display Controls

Do not blur these together.

Software color:

- implemented through gamma LUTs
- includes software brightness and software gamma
- lives in `src/cli/brightness.rs` and `Gamma` helpers
- preserves the existing compositor-installed LUT baseline when possible

Native display controls:

- power-save
- layout-mode
- backlight
- luminance
- color mode

Those are backend-native controls exposed by Mutter, not LUT tricks.

Maintenance rule:

- if a user asks for “brightness,” be very clear whether they mean software brightness or hardware/native brightness

## Rust Concepts You Actually Need Here

You do not need all of Rust to maintain this repo. You need a few ideas well.

### Ownership and Borrowing

The code uses borrowing heavily to avoid cloning large config structures.

Example pattern:

- `ApplyMonitor<'a>` borrows `&'a str` for connector and mode ids
- the planner builds borrowed apply payloads from the current `DisplayConfig`

This is efficient, but it means helper functions often need explicit lifetimes.

If you see a signature like this:

```rust
fn build_apply_configs<'a>(profile: &'a Profile, config: &'a DisplayConfig) -> ...
```

that usually means the returned structures borrow from `profile` and/or `config`.

When maintenance goes wrong here, it usually goes wrong as:

- temporary value dropped while borrowed
- borrow checker conflicts from trying to mutate and inspect the same structure in one expression

Fix strategy:

- introduce named locals
- shorten borrow scopes
- split expressions into smaller steps

### Enums as Typed State Machines

This repo uses enums for most “known set of values” domains:

- `LayoutMode`
- `ColorMode`
- `PowerSaveMode`
- `BrightnessFilter`
- `Reflection`
- `Rotation`

This is deliberate.

Do not replace typed enums with strings unless you have a very strong reason.

The enum layer is what keeps CLI parsing, query output, and D-Bus serialization from drifting apart.

### `Result` and Explicit Error Types

Most major modules define their own error enum.

That is good and should continue.

This repo benefits from errors that explain which layer failed:

- schema error
- capability mismatch
- backend rejection
- unsupported operation
- invalid user input

Maintenance rule:

- prefer explicit, typed errors over vague boxed strings in core logic
- it is okay to convert to `Box<dyn Error>` at CLI boundaries

### `bitflags`

`Transform` uses `bitflags`.

This is one of the more low-level parts of the codebase.

You recently already saw why it matters: newer `bitflags` versions changed API details such as `.bits()`.

Maintenance rule:

- when changing transform logic, verify both compile-time API expectations and semantic behavior
- do not assume a packed transform is the same thing as a high-level rotation or reflection

### `serde` and Schema Evolution

`query --json` and `apply FILE` depend on a stable, documented schema.

That means schema changes are maintenance-sensitive.

When you add fields:

- bump the schema version when needed
- keep older versions readable where practical
- update the README JSON section
- update `apply.rs` compatibility handling
- update tests

## The Main Code Paths

### 1. Query Path

Entry:

- `src/cli/mod.rs` -> `Command::Query`

Main logic:

- `src/cli/query.rs`

What it does:

- loads current `DisplayConfig`
- loads native display state where needed
- loads `Resources` where needed
- renders text or JSON

If you want to expose more state, `query.rs` is usually the first place you touch.

Maintenance rule:

- prefer typed fields first
- keep raw `properties` dumps available as supplemental data, not as the only way to discover important state

### 2. Modify Path

Entry:

- `src/cli/mod.rs` -> `Command::Modify`

Main logic:

- `src/cli/modify/mod.rs`

This is the highest-policy file in the repo.

It does four jobs:

1. parse and validate CLI combinations
2. resolve target hardware and capabilities
3. build or mutate a transactional layout plan
4. apply native controls and software color in the right order

If a feature is a layout feature, it usually ends up passing through the planner.

If it is a native control, it may apply after layout.

If it is software color, it applies after layout and after native display changes that may reset the ramp.

### 3. Planner Path

Main logic:

- `src/cli/modify/planner.rs`

This is the heart of multi-output layout application.

It operates on full `ApplyConfig` state, not single one-off mutations.

That choice is what enabled:

- disable/enable
- relative placement
- clone groups
- geometry reflow after rotation

Maintenance rule:

- if a feature changes topology, prefer putting the logic in the planner instead of bolting ad hoc mutation into `modify::handle()`

### 4. Software Color Path

Main logic:

- `src/cli/brightness.rs`
- `src/display_config/proxied_methods.rs` (`Gamma`, `GammaAdjustment`)

This is subtle and risky.

The important design is:

- do not reconstruct a simplified gamma curve from a live LUT
- preserve the current LUT as the baseline when external tools changed it
- only reuse a saved baseline when the current LUT still matches the last tool-managed state

This is what prevents Night Light / ICC / compositor color state from getting mangled.

If you break this logic, you will likely reintroduce old color-regression bugs.

### 5. Apply-From-File Path

Main logic:

- `src/cli/apply.rs`

This command reuses the documented `query --json` schema.

The important behavior is:

- match monitors by stable identity (`vendor`, `product`, `serial`)
- not by connector name alone
- rebuild one full apply payload
- replay managed software color after layout apply

Maintenance rule:

- if you change JSON schema or monitor identity assumptions, check `apply.rs` immediately

## Systems Programming Concepts That Matter Here

### You Are Driving Live Desktop State

This is not a pure data transformation repo.

Every meaningful command is acting on live desktop state through D-Bus.

That means:

- hardware may differ between runs
- current state may change between query and apply
- some backend methods exist but are only conditionally supported
- some properties report sentinels rather than usable values

Example:

- backlight properties can exist while current numeric values are unavailable or sentinel-coded

So the code must distinguish:

- supported
- unsupported
- present but unavailable
- valid but backend-rejected

### Capability Probing Is Part of Correctness

In this repo, correctness is not just “the code compiles.”

It is also:

- did we probe the backend capability first?
- did we select a typed surface GNOME actually supports?
- did we avoid pretending a backend limit is a normal CLI feature gap?

This is why the project often prefers explicit capability errors over fallback behavior.

### Transactional Thinking

`ApplyMonitorsConfig` wants a full logical monitor layout.

That means many changes should be planned transactionally, not applied incrementally.

If you mutate only one connector in isolation, it is easy to produce invalid layouts:

- overlap
- non-adjacent monitors
- broken clone groups

The planner exists to keep you from doing that.

## The Riskiest Maintenance Areas

If you only remember one section from this tutorial, remember this one.

### 1. `raw.rs` Is Autogenerated

Risk:

- hand edits get out of sync with the real D-Bus interface

Advice:

- keep hand edits minimal and compatibility-focused
- push logic outward into typed wrappers

### 2. Software Color Baseline Logic

Risk:

- reintroducing lossy LUT reconstruction bugs
- clobbering Night Light, ICC, or other compositor-installed color state

Advice:

- preserve the “current live LUT becomes baseline when state no longer matches saved managed state” rule
- do not simplify arbitrary LUTs into scalar parameters unless you are absolutely sure

### 3. Transform / Geometry Math

Risk:

- breaking relative placement
- breaking clone group geometry
- reintroducing non-adjacent layout failures

Advice:

- test both geometry shape and applied coordinates
- think in terms of logical monitor rectangles, not just single connector settings

### 4. Backend Limits vs CLI Gaps

Risk:

- spending time implementing fake support for things Mutter cannot actually represent cleanly

Advice:

- check the D-Bus interface and current GNOME tooling first
- if the backend is the limit, document it explicitly instead of inventing a misleading flag

### 5. Schema Drift

Risk:

- `query --json` and `apply FILE` silently diverge

Advice:

- treat schema changes as cross-cutting changes
- update parser, docs, tests, and preview logic together

### 6. Connector Names Are Not Stable Identity

Risk:

- restore/apply logic matches the wrong output

Advice:

- use `vendor`, `product`, and `serial` for stable identity
- treat connectors as current routing labels, not hardware identity

## How To Safely Change This Repo

When adding or modifying a feature, use this order:

1. Confirm the backend actually exposes the capability.
2. Add or update typed wrappers in `display_config/`.
3. Add or update CLI behavior in `src/cli/`.
4. Add dry-run output.
5. Add or update `query` visibility.
6. Update JSON schema and `apply.rs` if the saved layout format is affected.
7. Update README.
8. Update addressed notes if you are closing backlog/history work.
9. Run tests.
10. Run a live dry-run command.

Useful commands:

```bash
cargo fmt
cargo test
cargo run -- query --help
cargo run -- modify --help
cargo run -- query --json
cargo run -- modify --dry-run ...
cargo run -- apply /tmp/layout.json --dry-run
cargo install --path .
```

For shell integration checks:

```bash
bash -n <(cargo run --quiet -- completions bash)
zsh -n <(cargo run --quiet -- completions zsh)
cargo run --quiet -- completions fish >/dev/null
```

## How To Maintain It Going Forward

If you take over this repo, your job is not just to add flags.

Your job is to preserve the contract between:

- what GNOME/Mutter really supports
- what the CLI promises
- what the JSON schema documents
- what saved layouts can round-trip safely

The good maintenance posture for this repo is:

- typed over stringly
- explicit over magical
- backend-honest over xrandr-cosmetic
- transactional over incremental
- dry-run first
- docs updated with the code, not afterward

If you keep those rules, this repo is straightforward to maintain.

If you violate them, it will still compile, but it will become a desktop-state footgun.
